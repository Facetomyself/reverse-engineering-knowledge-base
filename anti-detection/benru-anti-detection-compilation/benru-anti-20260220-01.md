## 一、Session与Cookies：你的爬虫“身份证”

在深入代码之前，我们先来打个比方。

假设你要进入一个需要身份验证的  **神秘组织大楼**
（目标网站）。你第一次进去时，前台（服务器）会检查你的邀请函（账号密码），验证通过后，发给你一张  **临时通行证**
（Session），并在上面盖个章（Cookies）。之后你每进出一个房间（请求页面），只要出示这张盖了章的通行证，保安（服务器中间件）就放行。

在HTTP这个无状态的世界里，  **Session保存在服务器端** ，记录着“你是谁”；  **Cookies保存在客户端**
（你的爬虫），是通行证上的印章，用来证明你持有那张通行证。

爬虫登录状态管理，本质上就是  **如何保存、携带、更新这张“盖章通行证”** 。

## 二、四种主流的登录状态管理方案

### 方案A：基于  ` requests.Session  ` 的Cookie自动管理
![](https://mmbiz.qpic.cn/mmbiz_svg/Q3auHgzwzM6vWOSU3Tx4MicovYeS1s8CYupicCicxlVBdKns7kT7qicYpQpFCwkr7pEAicuANfLFFe7x2JpMeLguNEbDmAeO61DBHFIcttokdfwEWuTGUBtTEpw/640?wx_fmt=svg&from=appmsg)

这是最正统、最推荐新手入门的方式。  ` requests.Session  `
对象会自动保存服务器返回的Cookies，并在后续请求中自动携带，就像浏览器一样。

**优点** ：自动处理Cookies的存储和发送，代码简洁，逻辑清晰。
**缺点** ：Session对象只在程序运行期间有效，程序重启后Cookies丢失；不适合分布式爬虫。

    import requests

    # 创建一个Session对象
    session = requests.Session()

    # 模拟登录（以表单形式提交账号密码）
    login_url = 'https://example.com/login'
    login_data = {
        'username': 'your_username',
        'password': 'your_password'
    }

    # 发送登录请求
    response = session.post(login_url, data=login_data)

    # 检查登录是否成功（假设返回JSON包含状态）
    if response.json().get('status') == 'success':
        print("登录成功！Cookies已保存在session中。")

    # 后续请求自动携带Cookies
    profile_url = 'https://example.com/profile'
    profile_resp = session.get(profile_url)
    print(profile_resp.text)  # 打印个人页面内容

**适用场景** ：单机爬虫，一次登录，持续爬取，程序不中断。

### 方案B：直接复用Cookie字符串（从浏览器复制）
![](https://mmbiz.qpic.cn/mmbiz_svg/Q3auHgzwzM5l6gnibuIa2njMKZM4OxrxJyJ0FQEhjicgqOicTg1qmQibpQ5a7LOcRia24ockWia3yZjk02gurf8DuvNVbW13iaO1O23JUKOVT9WDxnsP455q0lJXQ/640?wx_fmt=svg&from=appmsg)

有时候，目标网站登录过程极其复杂——有验证码、有加密参数、甚至需要扫码。此时，手动在浏览器登录，然后直接把Cookie字符串复制出来，塞进爬虫里，是最省事的办法。

**优点** ：绕过复杂的登录逻辑，简单粗暴，适合一次性任务或反爬极强的网站。
**缺点** ：Cookie会过期；一旦失效，需要手动重新复制；不优雅，不适合自动化。

    import requests

    # 从浏览器开发者工具中复制到的Cookie字符串
    cookie_str = 'name=value; name2=value2; sessionid=abc123...'

    # 将字符串解析为字典
    cookies = {}
    for item in cookie_str.split('; '):
        key, value = item.split('=', 1)
        cookies[key] = value

    # 发起请求时携带Cookies
    url = 'https://example.com/user_info'
    resp = requests.get(url, cookies=cookies)
    print(resp.text)

> **注意** ：Cookie字符串中可能包含特殊字符，建议使用  ` http.cookies.SimpleCookie  ` 解析，更健壮。

**适用场景** ：临时调试、一次性数据抓取、登录环节无法自动化的网站。

### 方案C：Selenium/Playwright自动登录 + 导出Cookies给requests
![](https://mmbiz.qpic.cn/mmbiz_svg/Q3auHgzwzM4wdEcfwkuanSAMl92K4X6GMkAbVG2gfAib1Jx0NzW0yhuBiaSia9ROqGnuGDg2Z0slzdVibIKHJVKGhSOFl7zf72tcPXCOywem1Y8rj01hwKGY0w/640?wx_fmt=svg&from=appmsg)

浏览器自动化工具（Selenium、Playwright）可以模拟真实用户操作，解决验证码、JS加密等问题。但它们的爬取效率远不如  ` requests
` 。最佳实践是：  **用浏览器自动登录并获取Cookies，然后把Cookies交给  ` requests  ` 使用** 。

**优点** ：兼顾了自动化登录的灵活性和  ` requests  ` 的高效性。
**缺点** ：需要额外安装浏览器驱动，启动浏览器较慢，占用资源。

以Playwright为例（比Selenium更现代、更快）：

    import asyncio
    from playwright.async_api import async_playwright
    import requests

    asyncdefget_cookies_from_login():
        asyncwith async_playwright() as p:
            # 启动浏览器（无头模式可选）
            browser = await p.chromium.launch(headless=False)
            context = await browser.new_context()
            page = await context.new_page()

            # 访问登录页
            await page.goto('https://example.com/login')

            # 填写表单（这里根据实际情况调整选择器）
            await page.fill('input[name="username"]', 'your_username')
            await page.fill('input[name="password"]', 'your_password')
            await page.click('button[type="submit"]')

            # 等待登录成功，比如等待某个登录后才能看到的元素出现
            await page.wait_for_selector('.user-avatar', timeout=5000)

            # 获取当前上下文的Cookies（列表形式）
            cookies_list = await context.cookies()

            # 关闭浏览器
            await browser.close()

            # 转换为requests可用的字典格式
            cookies_dict = {cookie['name']: cookie['value'] for cookie in cookies_list}
            return cookies_dict

    # 异步运行获取cookies
    cookies = asyncio.run(get_cookies_from_login())

    # 交给requests使用
    session = requests.Session()
    session.cookies.update(cookies)
    resp = session.get('https://example.com/dashboard')
    print(resp.text)

**适用场景** ：登录环节复杂（验证码、扫码、JS加密），需要自动化获取最新Cookies的场景。

### 方案D：分布式登录状态共享（Redis集中存储）
![](https://mmbiz.qpic.cn/mmbiz_svg/Q3auHgzwzM5HCtt4Hvtib2oCvdgmQ49LiaKUV2QxO4PQcZWQpmD1mCf7TSLyfjZJfgNo7hnBb4N1D1cmcL5WjW6RV6UekJcMEfXgKCBYj2YKytNCrq0qQIkw/640?wx_fmt=svg&from=appmsg)

当爬虫规模扩大，部署在多台机器上时，每台机器各自维护登录状态会带来两个问题：

  1. 账号登录次数过多，触发网站风控（单账号多地登录）。
  2. 一台机器的Session过期了，其他机器不知道，依然用失效的Cookies请求，导致大量401/403。

解决方案：  **将登录状态（Cookies）集中存储在Redis中，所有爬虫节点从Redis读取最新的Cookies** 。

**优点** ：实现登录状态的统一管理和刷新，避免多机重复登录，降低封号风险。
**缺点** ：架构复杂，需要引入Redis，并处理并发读写问题。

    import redis
    import requests
    import json

    # 连接Redis
    r = redis.Redis(host='localhost', port=6379, db=0, decode_responses=True)

    defget_cookies_from_redis(account_key='cookies:account1'):
        """从Redis获取Cookies字典"""
        cookies_str = r.get(account_key)
        if cookies_str:
            return json.loads(cookies_str)
        returnNone

    defsave_cookies_to_redis(cookies_dict, account_key='cookies:account1', expire=3600):
        """保存Cookies到Redis，并设置过期时间（秒）"""
        r.setex(account_key, expire, json.dumps(cookies_dict))

    # 示例：先尝试从Redis获取
    cookies = get_cookies_from_redis()
    ifnot cookies:
        # 如果Redis中没有，则执行登录流程（可用方案A或C），拿到新的cookies_dict
        cookies = perform_login_and_get_cookies()
        save_cookies_to_redis(cookies, expire=3500)  # 设置略小于网站实际过期时间

    # 使用cookies发起请求
    resp = requests.get('https://example.com/data', cookies=cookies)

> **进阶**
> ：可以单独部署一个“登录维护服务”，定时检查Redis中的Cookies是否即将过期，如果即将过期，自动重新登录并更新Redis。爬虫节点只读不写，彻底解耦。

**适用场景** ：大型分布式爬虫系统、多账号轮询管理、需要高可用的生产环境。

## 三、避坑指南（老司机血泪经验）

### 1\. 混淆 Session 和 Cookies
![](https://mmbiz.qpic.cn/mmbiz_svg/Q3auHgzwzM6RY0iaicla4aTOPqjCHOLpx7iau5TMJl7ROIuAN0JvOBUJhvYt6Gic1f3wjeNlrjvujhtHPqUnHWdr2jF6jlhVHdAKkGjRvLLnQhCC7iaLbQySFzA/640?wx_fmt=svg&from=appmsg)

有些人认为有了  ` requests.Session  `
就万事大吉，但Session只是容器，真正的登录凭证是服务器下发的Cookies。如果程序重启后没有持久化Cookies，或者换了IP，Session还是空的，照样需要重新登录。

### 2\. HttpOnly 字段的影响
![](https://mmbiz.qpic.cn/mmbiz_svg/Q3auHgzwzM4PvPTuMLbTWaGFBIZV9Zcr6tUUia5LpUw9hqm753V75xPY8IIJw7OExK0JBXUkSKv0zqYlmMsWAyVQWh9MewxuSOfTL7bUBFRDbHiacCjBBlpw/640?wx_fmt=svg&from=appmsg)

浏览器中的Cookies可能带有  ` HttpOnly  ` 标志，意味着无法通过JavaScript获取（如  ` document.cookie  `
）。但没关系，我们的爬虫是在HTTP层面直接发送请求，  **只要在请求头中携带了这些Cookies，服务器就会认** ，  ` HttpOnly
` 只限制浏览器端脚本访问，对爬虫无影响。所以方案C中通过浏览器上下文获取的Cookies通常已经包含了所有需要的字段。

### 3\. 单账号多地登录的风控
![](https://mmbiz.qpic.cn/mmbiz_svg/Q3auHgzwzM4ulQ3fuvzpSjtB1TTXENdMBH2N3zNPuXGVtN4OjaLrbNYJZtBl5NvRdwu3dibSug9OJxav4leCHMEbklTANjMpNuTTHJnE1licPpDkxqAcH1mQ/640?wx_fmt=svg&from=appmsg)

网站会监控同一账号的登录IP。如果你把Cookies发给多个不同IP的爬虫节点，就可能触发“异地登录”警报，轻则踢下线，重则封号。解决方案：

  * 使用方案D的Redis共享模式，但只让一个节点持有Cookies，其他节点通过这个节点转发请求（代理模式）。
  * 或者使用账号池，每个账号只在固定IP段使用，减少风险。

## 四、总结与推荐

  * **小规模单机爬虫** ：优先选用  **方案A + Cookies持久化 + 自动重登机制** 。简单够用，维护成本低。
  * **登录极其复杂、偶尔爬取** ：方案B手动复制Cookie是最快的，但记得做好过期提醒。
  * **中等规模、需要自动化登录** ：方案C（Playwright/Selenium获取Cookies + requests爬取）是最佳组合。
  * **大型分布式系统** ：方案D（Redis集中管理+登录维护服务）是必选项。

在实际项目中，我往往将方案C和方案D结合：用Playwright维护一个“登录微服务”，定时刷新Cookies存入Redis，所有爬虫节点从Redis获取最新的登录状态。这样既绕过了复杂的登录反爬，又实现了高可用。

最后，送大家一句话：  **管理登录状态，就是在管理信任——让服务器每次看到你的请求，都像老朋友一样熟悉** 。
