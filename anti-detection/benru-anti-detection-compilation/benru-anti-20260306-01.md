爱它的人说：这是反爬界的艺术，用最优雅的方式挡住绝大部分爬虫。 恨它的人说：爬了一辈子数据，最后栽在几个“方块字”上。

猫眼电影、大众点评、天眼查……这些网站的数据在浏览器里看得清清楚楚，一爬下来全是乱码。你以为是编码问题？折腾半天GBK、UTF-8来回切，结果发现——人家压根没给你明文数据。

## 一、字体反爬到底是个啥？

### 1.1 现象：看得见，摸不着

打开猫眼电影，票房数字明明白白

但一扒源码，好家伙：

    <span class="stonefont">&#xf8fb;</span>
    <span class="stonefont">&#xf7a3;</span>
    <span class="stonefont">&#xf70c;</span>

复制到记事本，直接变方块。

这就是字体反爬的典型特征：  **页面显示正常，源码全是乱码** 。

### 1.2 原理：偷梁换柱

字体反爬的核心逻辑其实很简单：

网站不再直接给你“3.2亿”这个文本，而是给你一串特殊编码（比如  ` & #xf8fb  ;  `
），同时配一个自定义字体文件（.woff或.ttf）。在这个字体文件里，编码  ` & #xf8fb  ;  ` 对应的字形是数字“3” 。

浏览器拿到字体文件，渲染时把编码映射成对应的字形，你看到的就正常了。 爬虫只拿HTML，没拿字体文件，或者拿了也不会解析，看到的自然是一堆乱码。

### 1.3 技术演进：从静态到动态

字体反爬这些年也在“进化” ：

  * **阶段一（静态映射）** ：字体文件固定，同一个编码永远对应同一个数字。破解难度：★
  * **阶段二（动态加载）** ：字体URL带时间戳或随机数，每次请求换新文件，但字形不变。破解难度：★★
  * **阶段三（字形随机）** ：不仅文件URL动态，连字体轮廓坐标都随机变化，同一个数字在不同文件里长得不一样。破解难度：★★★★
  * **阶段四（复合干扰）** ：字体+CSS偏移+伪元素，一套组合拳。破解难度：★★★★★

## 二、破解方案一：坐标对比法（KNN机器学习）

对付“字形随机”这类高级反爬，需要上点机器学习的手段。

### 2.1 原理

每个字体文件里的字符，本质上是由一系列坐标点构成的轮廓 。就算字形看起来差不多，坐标数值也会有差异。但同一个数字（比如“3”）在不同字体文件里的坐标，
**相似度远高于它和其他数字的相似度** 。

基于这个原理，我们可以：

  1. 下载多套字体文件作为训练集
  2. 提取每个字符的坐标点作为特征
  3. 训练KNN分类器
  4. 对新字体文件里的字符进行预测，还原真实数字

### 2.2 核心代码实现

**第一步：下载字体文件，转成XML提取坐标**

    import requests
    from fontTools.ttLib import TTFont

    def download_and_parse_font(font_url, save_path):
        """下载woff字体并保存为xml"""
        resp = requests.get(font_url)
        with open(save_path, "wb") as f:
            f.write(resp.content)

        font = TTFont(save_path)
        xml_path = save_path.replace(".woff", ".xml")
        font.saveXML(xml_path)
        return font

**第二步：提取字符坐标**

    def extract_coordinates(font):
        """提取字体文件中每个字符的坐标点"""
        glyf_order = font.getGlyphOrder()[2:]  # 过滤前两个无用项
        coordinates = []

        for g in glyf_order:
            coors = font['glyf'][g].coordinates
            # 将坐标展平成一维列表
            flat_coors = [_ for c in coors for _ in c]
            coordinates.append(flat_coors)

        return coordinates, glyf_order

**第三步：训练KNN模型**

    from sklearn.model_selection import train_test_split
    from sklearn.neighbors import KNeighborsClassifier
    import pandas as pd
    import numpy as np

    def train_knn(training_data):
        """
        training_data: 多套字体文件的坐标数据，每套对应一组真实数字
        """
        # 假设training_data是列表，每个元素包含(坐标列表, 真实数字列表)
        X_list = []
        y_list = []

        for coors_list, true_nums in training_data:
            X_list.extend(coors_list)
            y_list.extend(true_nums)

        # 对齐维度（不同字符坐标数量可能不同，用0填充）
        max_len = max(len(x) for x in X_list)
        X_padded = [x + [0]*(max_len - len(x)) for x in X_list]

        X = np.array(X_padded)
        y = np.array(y_list)

        # 划分训练集测试集
        X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.25)

        # 训练KNN（参数1通常效果最好）
        knn = KNeighborsClassifier(n_neighbors=1)
        knn.fit(X_train, y_train)

        print(f"模型准确率: {knn.score(X_test, y_test)}")
        return knn, max_len

**第四步：预测并建立映射**

    def build_mapping(knn, target_font, max_len):
        """建立目标字体中编码与真实数字的映射字典"""
        coors_list, glyf_order = extract_coordinates(target_font)

        # 维度对齐
        X_predict = [x + [0]*(max_len - len(x)) for x in coors_list]
        X_predict = np.array(X_predict)

        predictions = knn.predict(X_predict)

        mapping = {}
        for i, uni in enumerate(glyf_order):
            # 转成HTML实体格式 &#x编码;
            html_entity = uni.lower().replace('uni', '&#x') + ';'
            mapping[html_entity] = str(predictions[i])

        return mapping

### 2.3 优缺点

**优点** ：能应付字形随机变化的情况，准确率高  **缺点** ：需要收集足够多的训练样本，每次字体大更新可能需要重新训练

## 三、破解方案二：OCR识别法（图片转文字）

如果说KNN是“硬核数学派”，那OCR就是“暴力美学派”。

### 3.1 原理

既然字体文件定义了每个字符的形状，那我们干脆把这些形状  **画成图片** ，然后用OCR识别图片里的内容 。

这个思路的牛逼之处在于：  **不管字体怎么随机变化，只要人眼能认出来，OCR就有可能认出来** 。

### 3.2 实现步骤

**第一步：字体转图片**

需要用到一个叫  ` fontforge  ` 的工具，它可以把字体文件里的每个字符导出为图片。

    # script.py - fontforge脚本
    import fontforge
    import os

    def ttf2pic(ttf_path, output_dir):
        """将ttf字体文件中的每个字符导出为图片"""
        font = fontforge.open(ttf_path)
        if not os.path.exists(output_dir):
            os.makedirs(output_dir)

        for name in font:
            # 每个字符单独导出
            glyph = font[name]
            if glyph.isWorthOutputting():
                # 设置图片尺寸
                pen = fontforge.pen()
                glyph.export(f"{output_dir}/{name}.png")

        font.close()

实际调用时，可以用子进程执行fontforge命令 ：

**第二步：OCR识别**

这里推荐用  ` ddddocr  ` ，轻量且准确率不错。

    import ddddocr
    import os
    from PIL import Image

    ocr = ddddocr.DdddOcr()

    def recognize_font_images(img_dir):
        """识别图片目录中的所有字符图片"""
        result = {}

        for img_file in os.listdir(img_dir):
            if not img_file.endswith('.png'):
                continue

            img_path = os.path.join(img_dir, img_file)
            with open(img_path, 'rb') as f:
                img_bytes = f.read()

            # OCR识别
            text = ocr.classification(img_bytes)

            # 文件名去掉.png就是字符编码
            code = img_file.replace('.png', '')
            result[code] = text

        return result

**第三步：建立映射表并替换**

    def decode_html_with_ocr(html_content, ocr_result):
        """用OCR结果解码HTML中的加密文字"""
        # ocr_result格式: {'uniE630': '3', 'uniE631': '8', ...}

        # 转换成HTML实体映射
        mapping = {}
        for code, num in ocr_result.items():
            if code.startswith('uni'):
                html_entity = code.lower().replace('uni', '&#x') + ';'
                mapping[html_entity] = num

        # 替换
        decoded_html = html_content
        for entity, num in mapping.items():
            decoded_html = decoded_html.replace(entity, num)

        return decoded_html, mapping

### 3.3 优缺点

**优点** ：通用性强，只要是字体反爬都能用这套方案，不依赖坐标特征  **缺点** ：速度慢（IO读写+OCR）、个别字形复杂时可能识别错误

## 四、两种方案对比

维度  |  KNN坐标对比法  |  OCR识别法
---|---|---
**适用场景** |  数字类、字体结构稳定  |  汉字/数字通用
**识别速度** |  快（毫秒级）  |  慢（秒级，含IO）
**准确率** |  高（95%+）  |  中（受图片质量影响）
**维护成本** |  需要定期更新训练集  |  基本无需维护
**实现难度** |  中等（需ML基础）  |  简单（调库即可）

**实战建议** ：

  * 数字类网站（猫眼票房）→ KNN坐标法
  * 汉字类网站（大众点评评价）→ OCR识别法
  * 混合型网站 → OCR兜底
