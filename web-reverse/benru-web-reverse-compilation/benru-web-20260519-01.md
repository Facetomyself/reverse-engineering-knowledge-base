> 比较手动补全、happy-dom、jsdom 与 vm2 四类 Node.js 补环境方案的覆盖面、隔离性、成本和适用边界。

你从网页里抠了一段加密代码，自信满满  ` node run.js  ` 。

啪——

    ReferenceError: window is not defined

你搜了搜，知道这叫"缺环境"。加上  ` globalThis.window = globalThis  ` ，再跑。

啪——

    TypeError: Cannot read properties of undefined (reading 'cookie')

加  ` document  ` 。加  ` navigator  ` 。加  ` location  ` 。加  ` btoa  ` 。加了一次又一次。

加密逻辑十几行。补环境的代码比加密本身还长。

每个搞 JS 逆向的人都会经历这个阶段。这不是你菜，是浏览器和 Node 本来就是两个世界——浏览器里躺着几十个全局对象，Node 默认一个都不给你。

问题变成：  **我怎么让这段代码在 Node 里跑起来？**

网上搜一圈，答案五花八门。有人说手动补、有人说上 jsdom、有人推荐 happy-dom、还有人提 vm2。

到底哪个靠谱？今天就把这些方案盘一遍。

* * *

## 补环境到底在补什么

先花 30 秒搞清楚本质。

浏览器里的 JS 能调用的 API，Node 里没有。最常见的：

    window         — 全局对象
    document       — DOM 操作
    navigator      — 浏览器信息（userAgent、platform）
    location       — URL 信息
    localStorage   — 本地存储
    btoa / atob    — Base64 编解码
    XMLHttpRequest — 发请求
    canvas         — 绘图（指纹检测）

你扣下来的加密函数可能调了其中几个。Node 不认这些 API，它就报错。

**补环境，就是在 Node 里把这些缺失的 API 模拟出来。**

就这么简单。不是什么高深技术。

但怎么模拟，有四种流派。

* * *

## 方案一：手补

最朴素的方式。缺啥补啥。

    globalThis.window = globalThis;
    globalThis.navigator = { userAgent: 'Mozilla/5.0 ...' };
    globalThis.document = { cookie: 'session=abc' };
    globalThis.btoa = (s) => Buffer.from(s).toString('base64');

代码写在你扣下来的加密函数前面。跑起来就不报错了。

**优点：** 零依赖、零学习成本、全盘可控。加密函数只调了 3 个 API，你手补 3 行就搞定。不用装包，不用学新 API。

**缺点：** 加密函数一复杂就崩。一个调了 30 个浏览器 API
的加密函数，你一个一个手补，补到一半可能就忘了前面补了什么。而且手补的环境是"假"的——  ` document.cookie  `
只是个字符串，不会联动更新。有些网站会检测这些细节。

**适合的场景：** 加密函数依赖的浏览器 API 不超过 10 个，或者你只需要跑通一次就跑。

**一句话总结：** 最原始但最可控。小场面用手补，干净利落。

* * *

## 方案二：happy-dom

如果加密函数调了十几个 DOM API，手补就有点遭不住了。

happy-dom 是瑞典开发者写的轻量 DOM 模拟库。

    const { Window } = require('happy-dom');
    const window = new Window();
    globalThis.window = window;
    globalThis.document = window.document;
    globalThis.navigator = window.navigator;

npm 装上，5 行代码。  ` document.cookie  ` 、  ` document.createElement  ` 、  `
window.location  ` 这些东西就都有了。

体积只有 3MB，首次加载 200ms。

**优点：** 比手补省事太多，比 jsdom 轻得多。大部分逆向场景它都够用。

**缺点：** 不是 100% 兼容完整浏览器。Canvas、WebGL 这些它不模拟。

**适合的场景：** 加密函数用了 10-30 个 DOM API，但逻辑不算特别复杂。

**一句话总结：** 逆向补环境的"甜区方案"。大多数情况选它没错。

* * *

## 方案三：jsdom

老牌选手，从 2010 年活到现在。

    const { JSDOM } = require('jsdom');
    const dom = new JSDOM('<!DOCTYPE html><html></html>', {
      url: 'https://example.com',
      userAgent: 'Mozilla/5.0 ...'
    });

**优点：** 几乎完整的 DOM 实现。改了  ` document.cookie  ` ，  ` window.document.cookie
` 也会同步。这不是模拟，这是真正在 Node 里跑了一个浏览器引擎。

**缺点：** 重。15MB，首次加载 800ms。而且它模拟的是"一个浏览器的子集"，不是浏览器本身——跑不了 Webpack 打包后的
Vue/React 应用。

**适合的场景：** 加密函数嵌了前端框架的运行时逻辑，依赖大量 DOM API 联动。

**一句话总结：** 大而全，但不是万能。能用 happy-dom 的时候别上它。

* * *

## 方案四：vm2（沙箱方案）

vm2 是一个 Node 沙箱——在一个隔离的 V8 虚拟机里运行不可信代码。

    const { VM } = require('vm2');
    const vm = new VM({ sandbox: { ... } });
    const result = vm.run('yourCode(param)');

**优点：** 代码隔离运行，不会污染宿主环境。你补的环境变量只在沙箱里生效，跑完自动销毁。适合团队协作——多个人补同一段加密，互不影响。

**缺点：** 沙箱有性能开销。而且沙箱不是铁桶——2023 年曝过逃逸漏洞，虽然已经修复了，但如果你在跑高安全场景，需要考虑这个风险。

**适合的场景：** 同一个脚本里要处理多个不同的加密函数，或者需要代码隔离的场景。

**一句话总结：** 隔离是它的核心价值，不是补环境本身——但隔离好的环境自然就不需要你自己拆了。

## 一张表看清楚

方案  |  体积  |  首次加载  |  DOM 完整度  |  最佳场景
---|---|---|---|---
手补  |  0  |  0ms  |  你补多少有多少  |  < 10 个 API，一次性的活
happy-dom  |  ~3MB  |  ~200ms  |  中等偏上  |  大多数逆向场景
jsdom  |  ~15MB  |  ~800ms  |  高  |  嵌了框架逻辑的复杂加密
vm2  |  ~3MB  |  ~100ms  |  隔离环境  |  多份加密同时处理，需要隔离

**附录：四种方案代码对比**

以下这段加密函数同时依赖三个浏览器 API：

    function generateSig(path, ts) {
        var ua = navigator.userAgent;
        var ck = document.cookie;
        var raw = path + '|' + ts + '|' + ua + '|' + ck;
        var hash = 0;
        for (var i = 0; i < raw.length; i++) {
            hash = ((hash << 5) - hash) + raw.charCodeAt(i);
            hash = hash & hash;
        }
        return btoa(hash.toString());
    }

**手补版**

    globalThis.window = globalThis;
    globalThis.navigator = { userAgent: 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36' };
    globalThis.document = { cookie: '' };
    globalThis.btoa = (s) => Buffer.from(s).toString('base64');

    // 然后直接调 generateSig

**happy-dom 版**

    const { Window } = require('happy-dom');
    const win = new Window({ url: '...', settings: { userAgent: '...' } });
    globalThis.window = win;
    globalThis.document = win.document;
    globalThis.navigator = win.navigator;
    globalThis.btoa = (s) => Buffer.from(s).toString('base64');

    // 然后调 generateSig

**jsdom 版**

    const { JSDOM } = require('jsdom');
    const dom = new JSDOM('<!DOCTYPE html>', { url: '...', userAgent: '...' });
    globalThis.window = dom.window;
    globalThis.document = dom.window.document;
    globalThis.navigator = dom.window.navigator;
    globalThis.btoa = (s) => Buffer.from(s).toString('base64');

    // 然后调 generateSig

**vm2 版（沙箱隔离）**

    const { VM } = require('vm2');
    const vm = new VM({
      sandbox: {
        navigator: { userAgent: 'Mozilla/5.0 ...' },
        document: { cookie: '' },
        btoa: (s) => Buffer.from(s).toString('base64')
      }
    });

    const result = vm.run(`
      function generateSig(path, ts) {
        var ua = navigator.userAgent;
        var ck = document.cookie;
        var raw = path + '|' + ts + '|' + ua + '|' + ck;
        var hash = 0;
        for (var i = 0; i < raw.length; i++) {
          hash = ((hash << 5) - hash) + raw.charCodeAt(i);
          hash = hash & hash;
        }
        return btoa(hash.toString());
      }
      generateSig('/api', '1680000000');
    `);

四种方案，同一个结果。选你合适的就行。
